UX Analysis - Slack

One of my assignments for SI 110 was to do a design analysis of a website, mobile application, or social media platform with respect to Jakob Nielsen’s 10 Heuristics for User Experience Design. I chose to review the desktop version of Slack.

As the pandemic necessitated heavy Slack use for school, I had curated a list of inadequacies (at least by my standards) in its interface and was eager to showcase what I’d learned thus far in the class. My main complaint was that it lacked a “landing page” for the user, as it instead constantly forces the user to be in one of their workspaces. This violates Nielsen's Third Heuristic, as it inhibits user freedom, control, and desire to explore the application. In other words, should the user lose their place in the interface or make a mistake, there are no spaces for the user to have a high-level overview of their workspaces nor are there any exits for them to “crash land” to. This decreases the user's confidence and makes them apprehensive about trying new capabilities of the application.

Among my other concerns were that some of the icons don’t represent what you would conventionally expect them to, the tendency for the screen to get cluttered with information, and that Slack seemed to cater towards power users rather than the average person.

Priority Queues, Pairing Heaps, and Video Games

Simulated a video game in which the player (the user) is supposed to choose the highest-priority enemy to defeat through the implementation of priority queues using custom comparators. Additionally, the player could specify output modes that would detail previous player actions and enemy statistics (median lifespan, longest-living enemies, etc.) by implementing the aforementioned data structures in other ways. The second half of this project was implementing a binary heap, unordered priority queue, and a pairing heap without the use of the C++ STL.

Zookeeper Pathway Optimization

Built a program that implemented a modified version of Prim’s algorithm to construct a minimum spanning tree while avoiding a predefined "danger zone" that could only be added to the tree if certain constraints were satisfied. Other capabilities of program included solving the Traveling Salesperson Problem through a branch-and-bound algorithm. The user could select for a more quickly retrieved yet suboptimal solution (nearest arbitrary insertion TSP heuristic).

Event Seating

At my cousin’s wedding reception this past July, I was looking around at all of the tables trying to figure out if there was any rhyme or reason to the seating arrangements. Some tables were completely full (or even overfilled), while others had plenty of vacant seats. When she came around to my family’s table, I asked my cousin how she had arranged all of us. She said that she and my new cousin-in-law had done the seating by hand, and that it was a massive pain to take into account the different groups of people as well as the number of chairs/tables available to them. Wanting to brush up on my Python skills, I decided that I’d make a tool for creating seating arrangements with groups manually inputted by the user. The user could also sort people in different ways, such as:

a) optimizing for a given number of chairs at each table
b) finding how many people should be seated with a given number of tables
c) having the program determine how many chairs/tables are needed based on the average group size
d) letting each larger-than-average group get its own table, even though this increases the amount of empty seats

Each of these options would print out what groups were at each table and who was in each group. Additionally, the program was designed so that the number of open seats at each table was minimized and so that as few groups as possible needed special accommodation (for each user preference).

Having furthered my education since creating this project, I am currently working on implementing disjoint sets to make sure that only people who know each other (e.g. bride’s family/groom’s family) are seated together.

Boredom Fighter

Like a lot of people, I hate being bored and feeling stagnant. Even when I have free time, I often face a lapse in creativity or decision paralysis where I feel like I just can’t find something to do. After describing this problem to a friend, they referenced this study about external reinforcement in decision making and how it helped them make their own decisions. Knowing this, I resolved to create an app that would TELL me what to do when I was bored rather than having to come up with something myself. Because it wasn’t my own idea to shoot down, it was harder to refuse.

Aside from the reasons above, the purpose of this project was to introduce myself to Swift and SwiftUI for iOS development.
